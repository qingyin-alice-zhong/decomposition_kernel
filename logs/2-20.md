Plan: Step-by-Step Decomposition Verification

 Context

 The current decomposition workspace asks agents to decompose a PyTorch model across all 4 levels (Model→Layer→Fusion→Kernel) in one shot, with only a final composition test for     
 verification. This has three problems:
 1. Errors compound — a wrong decomposition at Layer level silently propagates to all children
 2. No numerical verification with shared weights — the existing VGG16 composition test only checks shapes, not values
 3. No coverage guarantee — nothing automatically verifies all ops are accounted for

 The fix: decompose one level at a time, verify each step via a "refactored code" pattern (kernel falcon style), and use graph extraction for coverage analysis.

 Files to Create

 1. scripts/verify_step.py — Step-Level Numerical Verification

 The core verification engine. Given an original component and a refactored version that calls child modules:
 - Loads both models from .py files
 - Builds a weight map (explicit JSON, or auto-matched by shape)
 - Transfers weights from original → refactored
 - Runs both on multiple random inputs (3 trials, different seeds)
 - Compares outputs with dtype-aware tolerance (fp32: rtol=1e-5/atol=1e-6, bf16: rtol=1e-3/atol=1e-3)
 - Reports PASS/FAIL with max_diff, unmapped keys, per-trial details
 - Outputs verification_result.json

 2. scripts/extract_ops.py — Coverage Analysis via Graph Extraction

 Three-tier fallback strategy to extract all ops from a model:
 1. torch.fx.symbolic_trace + ShapeProp — works for standard models (VGG, DenseNet)
 2. torch.compile with capture backend — handles control flow (ResNet's if downsample)
 3. Forward hooks on leaf modules — always works, but misses functional ops (F.relu, torch.matmul)

 Compares extracted ops against decomposed children. Reports: total ops, covered ops, missing ops, extra ops, coverage %.

 3. verification/step_refactored_template.py — Template for Refactored Code

 Template the agent fills in at each step. Contains:
 - RefactoredModel class that imports and calls child modules
 - get_weight_map() function mapping original param names → refactored param names
 - get_inputs() matching the parent's inputs
 - Self-contained verification runner

 4. scripts/run_step_pipeline.py — Orchestrator

 Post-hoc validation tool that reads decomposition_tree.json, identifies all parent→child edges, and runs verify_step.py + extract_ops.py for each edge. Produces
 step_verification_summary.json.

 The Refactored Code Pattern (Key Concept)

 At each step, the agent takes the parent component and rewrites its forward() to call child modules instead of inline computation. Example:

 Original Block1: Conv2d → ReLU → Conv2d → ReLU → MaxPool (all inline in Sequential)

 Refactored Block1:
 from children.conv_relu_3x64 import Model as ConvRelu1
 from children.conv_relu_64x64 import Model as ConvRelu2
 from children.maxpool2d import Model as MaxPool

 class RefactoredModel(nn.Module):
     def __init__(self):
         super().__init__()
         self.conv_relu_1 = ConvRelu1()
         self.conv_relu_2 = ConvRelu2()
         self.maxpool = MaxPool()
     def forward(self, x):
         x = self.conv_relu_1(x)
         x = self.conv_relu_2(x)
         x = self.maxpool(x)
         return x

 Verification: Load original weights, transfer via weight_map, run both on same input, assert torch.allclose.

 Edge cases handled in refactored code:
 - Residual connections stay in the refactored forward() (not pushed into children)
 - Multiple outputs: children propagate tuples, refactored code unpacks
 - Shared weights: weight_map maps one original key to multiple refactored keys

 Plan: Step-by-Step Decomposition Verification

 Context

 with only a final composition test for verification. This has three problems:
 1. Errors compound — a wrong decomposition at Layer level silently propagates to all children
 2. No numerical verification with shared weights — the existing VGG16 composition test only checks shapes, not values
 3. No coverage guarantee — nothing automatically verifies all ops are accounted for

 Script Usage Model

 Primary path: The agent calls verify_step.py directly as a bash command after each decomposition step. The script is a standard tool the      
 agent uses, not just a reference.

 Fallback: If a model has special components the standard script can't handle, the agent writes a custom verification script that replicates   
 the exact same logic (same tolerance rules, same weight transfer, same output format) as verify_step.py.

 Anti-cheat enforcement: The standard script validates that the refactored code doesn't duplicate the original's computation — it must ONLY    
 call child subgraph modules plus allowed data plumbing.

 Files to Create

 1. scripts/verify_step.py — Standard Step Verification Script (agent calls directly)

 The canonical verification tool. The agent runs it as:
 python scripts/verify_step.py \
     --original path/to/parent.py \
     --refactored path/to/refactored.py \
     --children-dir path/to/children/ \
     --output path/to/verification_result.json

 What it does:
 1. Loads original model and refactored model from .py files
 2. Anti-cheat checks on refactored code:
   - All nn.Parameters in RefactoredModel must belong to child submodules (no standalone weights)
   - Scans forward() source for disallowed patterns: nn.Linear, nn.Conv2d, F.linear, F.conv2d, torch.matmul etc. — any parameterized or        
 compute-heavy op that should be in a child
   - Allowed in refactored forward(): child module calls, tensor routing (+, torch.cat, torch.split, indexing), shape ops (reshape, permute,   
 transpose, view, flatten, unsqueeze)
   - Reports violations before running numerical comparison
 3. Builds weight map: auto-matches by name, then by shape+dtype, or reads explicit weight_map.json
 4. Transfers weights from original -> refactored
 5. Runs both on same inputs (3 trials, different seeds)
 6. Compares outputs with dtype-aware tolerance:
   - float32: rtol=1e-5, atol=1e-6
   - float16/bfloat16: rtol=1e-3, atol=1e-3
   - int types: exact match
 7. Reports PASS/FAIL with max_diff, unmapped keys, per-trial details
 8. Outputs verification_result.json

 Anti-cheat detail — allowed vs disallowed operations in refactored forward():

 ┌──────────────────────────────────┬──────────────────────────────────────────┐
 │     Allowed (data plumbing)      │      Disallowed (must be in child)       │
 ├──────────────────────────────────┼──────────────────────────────────────────┤
 │ self.child_module(x)             │ nn.Linear(...), nn.Conv2d(...)           │
 ├──────────────────────────────────┼──────────────────────────────────────────┤
 │ x + residual, x * scale          │ F.linear(...), F.softmax(...)            │
 ├──────────────────────────────────┼──────────────────────────────────────────┤
 │ torch.cat(...), torch.split(...) │ torch.matmul(...), torch.bmm(...)        │
 ├──────────────────────────────────┼──────────────────────────────────────────┤
 │ x.reshape(...), x.permute(...)   │ F.gelu(...), F.relu(...)                 │
 ├──────────────────────────────────┼──────────────────────────────────────────┤
 │ x.transpose(...), x.view(...)    │ F.layer_norm(...), F.batch_norm(...)     │
 ├──────────────────────────────────┼──────────────────────────────────────────┤
 │ x[:, :, 0] (indexing/slicing)    │ Any nn.Module not imported from children │
 └──────────────────────────────────┴──────────────────────────────────────────┘
